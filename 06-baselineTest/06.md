## 评估和基准测试

评估核心的问题是给定一个固定的模型，评判其到底有多好，不仅决定了我们如何衡量当前模型的性能，更在根本上塑造了未来模型的发展方向。

### 简介

打开LLM的评测网络或者论文时的测试数据，主要包含以下的几个方面

1. 基准测试分数Benchmark Scores

   这是最直观、最普遍的评估形式，在各个大模型发布时都会在一系列标准化的基准上报告得分，这些标准化的基准主要包括MMLU、GPQA、MATH 500、Codeforces等。大部分语言模型都会在大致相同的基准上进行评估，但是它们不是完全一致的。

   国内也有一些大模型的榜单OpenCompass（上海AI实验室2023年8月推出的大模型开放评测体系）、SuperCLUE（国内第三方AI评测分析机构）等

2. 成本和性价比

   评估时的成本和推理速度也是评价的关键维度，比如Artificial Analysis网站从智能、推理速度、价格三个角度来评测模型。将模型的性能和每token的成本结合起来绘制出帕累托前沿Pareto Frontier，揭示了如下事实：顶级的模型（比如O3）很强大，但是价格昂贵，而一些排名稍靠后的模型可能在性能和成本之间取得了更好的平衡

3. 用户选择和市场反馈

   另外一种评估是看用户实际选择谁，OpenRouter通过路由流量到不同的模型，积累了大量关于用户偏好的数据，根据发送到每个模型的token数量生成了一个使用量排行，“被广泛使用”本身也是一个“好”的指标

   Chatbot Arena也是一个很受欢迎的评测榜单，由国际开放研究组织 LMYSY Org推出的大模型竞技评测平台，最大特点是采用**匿名盲测机制**随机配对模型，通过用户的投票和Elo评分系统量化模型能力，支持多轮对话和针对性榜单测评。

4. 主观评价和社区口碑

   来自社交媒体的使用者的感觉、评价和分析也是评估模型能力的侧面来源

许多主流的标准如MMLU等，之前看起来不错，但是现在人们认为要么已经饱和，要么被刷榜，或者两者兼有，Chatbot Arena因商业利益受到质疑，现在手握数据和排名但是缺乏一套清晰、可靠、公认的评估标准

### 如何看待评估

评估不是一个机械化的脚本，是一个需要深思熟虑的框架，必须要服务一个明确目标

核心原则是“**没有唯一的正确评估**”，评估的意义取决于想回答的问题：

- 对于终端用户/企业：我应该采购哪个模型来满足我的具体需求？
- 对于研究人员：我们是否在模型的原始能力（如“智能”）上取得了科学进步？
- 对于政策制定者/安全机构：当前模型带来的益处和风险分别是什么？现状如何？
- 对于模型开发者：我该如何改进我的模型？哪些干预措施有效？

把抽象目标转化为具体评估方案的四个关键环节

1. 输入是什么？
   - 提示词的来源，覆盖了哪些使用场景？
   - 是否在长尾分布中包含了具有代表性的困难输入样本？（长尾分布指出现频率很低、不常见但是真实存在的输入情况，比如在语言模型中可能包括的罕见领域的问题、复杂的推理任务、边缘语言现象、对抗性或者歧义性强的语句等）
   - 输入是否适配模型？
2. 如何调用模型
   - 如何提示模型？（零样本/少样本/思维链CoT）
   - 是否允许模型使用外部工具？（如代码解释器、RAG检索增强生成）
   - 在评估模型本身还是评估一个由模型和代理框架组成的完整系统？（开发者关心前者，用户关心后者）
3. 如何评估输出？
   - 参考答案是否准确无误
   - 使用的指标是什么？对于代码生成，使用pass@1还是pass@10（pass@n代表模型生成n个回答，n个输出回答中有一个和标准答案一样即为正确）
   - 如何考量成本？（更大但是提升效果有限的模型是否值得高推理延迟和训练开销？，**在资源受限场景的关键约束是成本**）
   - 如何处理不对称错误？（在医疗、金融、法律等高风险领域中的幻觉的代价远高于其他的错误）
   - 如何评估开放域生成任务？（没有标准答案时的评估很困难）
     - 使用人工评估（成本高、可靠）
     - 用 LLM-as-a-judge（让另一个大模型打分）
     - 基于多样性、流畅性、相关性等维度设计 Proxy metrics（代理指标，是指在无法直接测量目标量（target quantity）时，用来间接反映或近似该目标的可测量指标）
     - 通过用户反馈或 A/B 测试衡量实际效用
4. 如何解读结果？
   - 91%的分数意味什么，是否可以部署给用户？
   - 如何判断模型是否真正掌握了某种泛化能力？
   - 到底在评估什么？（产品模型/完整系统/研究方法）

### 困惑度 Perplexity

语言模型本质是概率分布$p(x)$，对任意的一段token序列x给出一个概率值——序列有多大概率被模型生成。困惑度衡量了模型对某个数据集分配高概率的能力，在预训练阶段，模型的目标是最小化训练集上的困惑度，数值越小表示模型认为这些数据“很自然”，预测能力强。
$$
Perplexity=(\frac{1}{p(D)})^{1/N}
$$
$p(D)$是模型给整个数据集D中所有句子联合概率的乘积（通常取log求和后再指数还原）；$|D|$是数据集中token数量

**使用测试集原因**

再预训练阶段通过最小化训练集上的困惑度优化模型参数，但是为了评估模型是否真的学会了语言，必须在没有见过的数据上测量困惑度，这才是模型泛化能力的体现

在GPT-2论文中展示了不同规模模型在多个任务上的表现，得出下面的结论：

- 随着模型变大，困惑度持续下降——更大的模型能更好的拟合语言分布
- 在某些任务上准确率也随模型增大而提高——语言建模能力迁移到下游任务也是有利的
- 但是在1BW上大模型的改进幅度较小——该数据集难度更高或者已经接近极限

**困惑度的重要性**

1. 更平滑，适合拟合Scaling laws，下游任务的准确率往往波动大，且有天花板，而困惑度变化连续，便于研究模型性能如何随着参数量/数据量增长
2. 通用性强，适用任何文本，更适合作为基础能力的度量
3. 可以扩展到条件困惑度，在问答任务中，固定问题部分只计算答案部分的困惑度，也反映了模型在该任务上的语言建模能力

**适用困惑度的陷阱**

评估困惑度则要求模型必须输出每个 token 的概率分布，并且这些概率加起来必须等于 1（归一化）。这依赖于模型内部机制，存在信任问题：

- 模型偷偷作弊（比如输出虚假概率）时无法检测
- 早期模型遇到未知词UNK时处理不当会导致概率不准确

困惑度更适合研究者自己控制模型架构和训练过程的场景，不适合完全黑盒的公开评测。

**一些可以反映模型是否真正理解语言的下游任务**

1. 完形填空类任务：给定一段上下文，让模型预测其中一个被遮盖的词，这个任务要求模型不仅要理解局部语法，还要捕捉长距离的依赖、语意连贯性、世界知识等，只关注某个关键位置的预测质量而不是整个句子，完形填空可以看作局部困惑度
2. 常识推理任务：给定一个视频片段或者一段文字描述premise，然后给出四个选项要求选出最符合常识、最自然、最可能发生的后续动作，强调日常生活的常识推理和行为合理性判断。HellaSwag（常见的常识推理任务）可以看作是**情境下的困惑度**，模型不需要输出概率，但是它的选择应该反映出它对哪个结果最可能的内在概率估计

### 知识类基准

1. MMLU Massive Multitask Language Understanding：包含57个学科的多项选择题，由学生收集，更侧重知识而非语言理解。Gemini 3 Pro Preview已达到90.3%的准确率
2. MMLU-Pro：对MMLU的改进，移除了噪声大和过于简单的问题，把选项从4个增加到10个，模型的准确率显著下降，缓解了饱和问题，通常结合思维链CoT评估
3. GPQA Graduate-Level Google-Proof Q&A：由61名PhD通过upwork平台设计的高难度问题，目标是创建”防谷歌“问题，非专家即使花费30min用谷歌搜索也难以解答
   - 博士级专家实现65%准确率
   - 非专家在访问谷歌的情况下30min内达到34%准确率
   - GPT-4准确率39%，Gemini 3 Pro Preview达到80.3%
4. Humanity's Last Exam：包含2500道多模态、多学科的选择题和简答题，用前沿 LLM 筛选掉过于简单的问题，再经过多阶段审查。局限在于问题征集过程可能存在严重的选择偏差，且问题类型仍局限于有标准答案的“考试”形式。Gemini 3 Pro Preview达到38.3%准确率

### 指令遵循基准

这类的基准评估模型是否能按照用户的要求进行输出

1. Chatbot Arena：采用“盲测”和 ELO 评分系统。真实用户提交提示，同时收到两个匿名模型的回复，并选择更优者。优点是输入动态、能容纳新模型。问题在于评估者是网站访客，样本可能存在偏差；ELO 分数可能被策略性操纵。
2. IFEval Instruction-Following Eval：使用可自动验证的约束（回答必须包含至少5句话）测试模型。优点是自动化程度高。局限是只评估约束遵守情况，不评估语义质量，且约束本身可能过于人工化。
3. AlpacaEval：包括各种来源的 805 条指令。使用一个强大的 LLM（如 GPT-4）作为裁判，判断候选模型的回答是否优于 GPT-4 自身的回答，并计算胜率。优点是自动化程度高，能处理开放域回答。问题在于存在裁判偏见，且早期版本易被回答长度等表面特征欺骗。
4. WildBench：从100多万条真实人机对话中采样构建评估集，并使用 GPT-4-Turbo 作为裁判，配合检查清单确保评估维度的全面性。其结果与 Chatbot Arena 高度相关（相关系数 0.95），被视为新基准有效性的“事实上的”检验标准。

### 智能体基准

这类基准评估模型作为Agent的能力，在复杂环境中通过工具调用和迭代规划完成任务

1. SWEBench：包含 12 个 Python 代码库中的 2294 个任务，根据 GitHub issue 描述，提交一个能通过单元测试的 Pull Request (PR)。评估直接运行单元测试来验证修复是否成功。
2. CyBench：完成 40 个网络安全领域的“夺旗”（CTF）挑战。任务难度通过人类“首次解决时间”来度量，有些任务对人类而言也极具挑战性（耗时可达24小时）。
3. MLEBench：自动化参与 75 个 Kaggle 机器学习竞赛，包括数据处理、模型训练、超参调优和结果提交。当前最佳模型获得任何奖牌的准确率也低于20%。

有些基准代表了LLM能力的最前沿，但是当前模型的成功率普遍较低，表明距离真正的自主智能体仍有巨大差距

### 纯推理基准

这类基准试图剥离知识，仅评估模型抽象推理能力

1. ARC-AGI   Abstraction and Reasoning Corpus for Artificial General Intelligence：提供一系列视觉化的输入-输出网格对，要求模型推断出变换规则并且应用于新的输入，整个过程不涉及任何语言

   它捕捉了一种更纯粹的、类似人类的模式识别和泛化能力，是早期 AGI 研究的重要基准。传统 LLM 在此任务上表现极差，但最新模型已展现出一定能力。

### 安全基准

安全评估旨在衡量模型执行任何有害行为的倾向。

1. HarmBench：定义了510种违反法律或者社会规范的有害行为，通过提示词测试模型是否会执行，评估其拒绝率。
2. AIR-Bench：基于全球监管框架和公司政策，构建了一个包含314个风险类别和5694个提示的系统性评测集
3. Jailbreaking越狱：即使模型被训练为拒绝有害请求，攻击者也可以通过自动优化提示绕过安全防护（比如Greedy Coordinate Gradient，GCG）。这类攻击甚至能从开源模型迁移到闭源模型。[2307.15043](https://arxiv.org/pdf/2307.15043)
4. 部署前测试：美国安全研究所和英国人工智能安全研究所携手合作，公司在发布前向安全机构提供模型访问权限（目前为自愿性质），安全机构进行评估并向公司提交报告
5. 安全也反映了模型的能力，在医疗等高风险场景，减少幻觉本身就是提升安全性和能力，对于闭源API模型，倾向性是关键，对于开源基础模型，能力本身就是风险

### 真实性

语言模型在实践中被广泛应用，大多数的基准测试和现实的使用场景相差甚远，并且真实用户的提示大多是**求助型**而非**测试型**

1. Clio（Anthropic）：提出使用语言模型分析真实用户数据分享人们提问的一般模式
2. MedHELM：由29个临床医生贡献121个真实的临床任务，更贴近实际医疗应用场景。然而，真实性与隐私之间存在天然矛盾，许多真实数据（如患者病历）无法公开。

### 有效性

评估有效性有两个挑战

1. 训练-测试集重叠 Train-Test Overlap：在预训练数据即整个互联网的时代，确保测试集没被模型见过变得十分困难，导致评估结果虚高，解决方法包括开发工具推断重叠情况和推动行业规范，要求模型提供商主动报告重叠检测结果。
   - 尝试从模型中推断训练集和测试集的重叠部分：利用数据点的可交换性尝试从模型中推断训练集和测试集的重叠部分
   - 鼓励报告规范：提倡模型提供者应报告训练集与测试集的重叠情况
2. 数据集质量：许多基准存在标注错误或噪声，修正后的SWE-Bench版本表示原始分数可能因为错误而被低估。

### 评估的对象

在ImageNet时代更多评估的是方法，在固定数据集和训练协议下，新算法的优劣

现在评估的更多是模型/系统，“端到端”的最终产品，开发者可以使用任何数据和技巧。

评估方法可以促进算法创新，评估系统对下游用户更有意义，两种范式都有价值，但是要清晰界定评估的规则，否则会失去比较的意义。
