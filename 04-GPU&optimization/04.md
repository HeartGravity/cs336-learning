## GPU及其相关的优化

在深度学习中硬件的性能和对并行化更好的利用决定了训练和推理模型的性能。

在硬件规格的提升过程中，还需要做到计算效率的扩展，早期使用的是Dennard 缩放定律，随着晶体管的面积的缩小，芯片上的晶体管数量越来越多，计算的效率变高的同时降低能耗，综合提升芯片的性能，但是在2000年代，随着晶体管密度增大，单线程计算的效率表现并没有如期增长，增长的趋势逐渐变缓。

提升芯片性能的方式转向使用并行计算，并行缩放定律成为主流，GPU的并行缩放定律使得其性能在十几年内提升了超过1000倍，为后续的LLM中的缩放定律提供了基础。

### GPU的起源——图形处理器

计算机中渲染的3D模型是由一个个小的三角形组成的，三角形由三根线构成，为了节省存储的空间只存储三角形的三个顶点坐标，构成线的这些像素点实时计算出来并不存储。

实时计算的过程主要是通过顶点坐标计算斜率和截距进而求得剩下的像素点坐标，涉及了大量的简单的乘法和加法，但是使用CPU做计算时由于只能串行处理导致计算的时间非常长，所以需要一个**大量并行计算简单的乘法和加法**的计算单元，就是GPU。其中GPU的计算单元为**CUDA核心**。

**CPU和GPU的区别**

在CPU中程序按顺序串行运行，单线程中逐步执行指令，由于存在大量的分支和条件控制的逻辑，CPU将大量的芯片面积用于分支预测，所以控制单元的面积占了很大一部分，核心数量有限但是运行速度快。但是GPU中大部分的空间都留给计算单元，GPU中有海量的计算单元，但是控制单元的部分只占一小部分，用于编排和协调计算单元的并行。

从两者的设计目标出发，CPU的设计目标是追求单个任务快速完成，优化延迟，但是GPU的设计目标是追求整体任务快速完成，优化吞吐量。GPU中有大量可以快速休眠随后快速唤醒的线程，虽然GPU每个任务的延迟很高，但是整体的完成时间少于CPU，GPU的架构不同在于GPU会运行大量的**流式多处理器(SM)**。

GPU的控制单元和缓存部分决定了可以完成复杂的逻辑运行，GPU的大量计算单元决定了可以进行大量并行的简单的乘法加法运算，但是无法处理复杂的逻辑计算。在使用GPU时还需要CPU的调度，所以一个好的GPU要配上好的CPU才能发挥作用。

在AI时代中神经网络的计算涉及到了大量的矩阵运算，其本质是大量的乘法和加法，这时GPU就适合来做这些简单重复的运算，在2010年左右人们开始使用GPU进行AI相关的计算。

**GPU的发展**

1. 1980年代前：电脑显示全靠CPU计算，显示的颜色种类很少，在1985年ATi公司开始使用ASIC技术做图形芯片，1992年该公司的Mach32图形卡首次集成图形加速功能，被看作GPU的“雏形”
2. 1990年代：没有GPU概念，称为图形加速器，逐渐诞生3D的加速芯片，但是各家的标准混乱互不兼容，只能处理一些特定的3D任务，功能专一
3. 2000年左右：NVIDIA发布了GeForce 256，首次提出GPU的概念，在硬件T&L技术上把3D图形的坐标变换、光照计算从CPU解放出来，变成GPU专职，同时实现了单芯片集成，整合三角形构成、裁剪、纹理、渲染功能，实现性能飞跃，大幅减轻CPU的3D计算负担。
4. 2001-2012可编程时代：2001年微软DirectX 8 引入顶点着色器和像素着色器，GPU可以跑简单的程序；2006年NVIDIA发布GeForce 8800 GTX首个统一渲染架构GPU，同时发布CUDA技术让GPU可以跑C语言程序。
5. 通用计算时代2012+：AI研究人员开始使用GPU训练深度神经网络，NVIDIA构建的CUDA生态让程序员可以轻松的调用GPU算力

**A100显卡核心的构成**

显卡由供电、显卡核心、显存、显示接口和金手指组成。

显卡核心由cuda core、控制单元和缓存单元等组成，由于GPU负责的大多是重复性的3D建模或者渲染，流处理器负责顶点计算或像素计算，动态分配进行顶点运算和像素运算的流处理器数量达到资源高效利用。

HBM2e显存堆栈不同于消费级GPU的GDDR显存颗粒，A100使用3D堆叠技术，共8层显存堆栈，每层8GB容量，通过硅通孔垂直互联。

A100总共有四个层级的架构拓扑，主要为 GPC->TPC->SM->CUDA核心

一个显卡核心有8个（实际中启用7-8个）GPC（图形处理簇），一个GPC有12个TPC（纹理处理簇），总共96个TPC，每个TPC有2个SM流式多处理器，总共有192个流式多处理器（实际启用了108个），每个SM有64个CUDA核心，一共有6912个CUDA核心。每个SM有192个Tensor core。

Tensor core的核心技术：结构化稀疏（自动跳过0值计算，有效性能提升2倍），多精度融合（单周期内处理不同精度的数据）

### GPU的执行模型SM

**SM核心作用**

SM可被视为一个最小单元，使用Triton编程时操作层级对应SM，每个SM内包括多个流处理器SP，每个流处理器并行执行大量线程，SM作为控制器决定执行内容，比如实现分支的判断，SP负责将相同的指令应用到不同的数据片段，实现海量的并行计算。在这种架构下每个SM是控制粒度的基本单元，单个SP可以独立完成大量计算。每个SM内部集成大量SP和专用的矩阵乘法单元，每个SM能操控专属组件（如张量核心）进行计算

- 线程调度和执行：SM同时管理数千个线程，决定哪个线程在何时使用哪个计算单元，不为线程保存大量状态，而是轻量级的切换，几乎没有开销
- 指令流水线：SM内部有4条独立的指令流水线，每个时钟周期可以同时发射4条不同指令给不同的Warp（线程束）
- 数据缓存和共享：SM内置了192KB的L1缓存/共享内存，供本SM内所有的CUDA核心快速存取数据，延迟比全局的显存低100倍。

### 执行模型中的核心名词

执行模型有三个重要的参与者

- 线程：线程并行“执行工作”——所有的线程执行相同的指令但是输入不同（SIMT）
- 块：线程的组，每个块在一个SM上运行，拥有自己的共享内存
- 束：线程总是以32个连续编号的线程组成的“束”执行。

由块到线程束到线程，粒度逐渐细化。每个SM是独立工作的单元，块就是分配给他的处理单元。基本的执行模型为：多个块被分配给不同的SM，每个块内包含多个线程束，每个线程束又由大量线程组成，所有的这些线程会在不同的数据上执行相同的指令。

![image-20260424202742762](C:\Users\32109\AppData\Roaming\Typora\typora-user-images\image-20260424202742762.png)

1. 线程束Warp：一个Warp是32个线程组成的固定小组，是SM调度的最小单元，Warp中的所有线程同时启动同时结束，执行完全相同的指令，如果某个线程需要走不同的分支，其他的线程需要等待，称为线程束分化 Warp Divergence。SM同时驻留64个Warp，4个Warp调度器每个管理16个Warp，Warp内的32线程可以在SIMD单元上同步执行
2. 线程块Block：Block映射在一个SM上执行，块内共享内存，每个Block独占SM的共享内存和寄存器资源，Block中所有线程必须在同一SM内执行（不能跨SM执行）
3. 线程Thread：最细粒度的执行单元，每个线程执行同样的Kernel代码，但是操作不同的数据，每个线程负责一个数据元素，每个线程有私有寄存器（通常为256个），线程ID决定它处理哪个数据
4. 单指令多线程SIMT：GPU执行模型，多个线程Warp共享同一条指令但是操作不同的数据。

![image-20260424204317252](C:\Users\32109\AppData\Roaming\Typora\typora-user-images\image-20260424204317252.png)

内存距离SM越近访问的速度越快，因此存在极高速的内存类型（L1缓存和共享内存），它们位于SM中有着极快的读写速度，寄存器这类需要频繁读写的元件适合放在L1和共享内存中。GPU芯片旁边安装了DRAM内存，意味着数据需要实际离开芯片通过物理连接进行传输，芯片上边缘的HBM连接器连接到实际GPU外部的DRAM芯片。

全局内存和L2缓存和内部的高速内存将近10倍的速度差使得如果某段计算需要访问全局内存，则SM会无工作可做，任务耗尽只能空转导致利用率降低。

代码可以写入全局的内存，每个线程都能访问自己的寄存器和共享内存，跨线程块的信息需要写入全局内存。当编写执行任务的线程时理想状态下它们应该操作相同的小批量数据，可以将这一部分小批量数据加载到共享内存中，所有的线程都能高效的访问共享内存，执行完毕后任务完成。反之如果线程需要到处访问数据就必须访问全局内存，速度会非常慢。

1. 第一层：全局内存

   - 物理位置：GPU芯片外的HBM2e显存堆栈
   - 容量：A100：40GB/80GB
   - 带宽：2039GB/s（A100）
   - 延迟：约为500 GPU周期（约250ns）
   - 编程控制：手动管理(cudaMalloc)
   - 可见性：所有线程可访问

   全局内存可以存放模型的权重、激活值、梯度（如GPT-3的1750亿参数）；训练数据、中间结果、最终输出，将数据持久化，通过PCIe从主机内存拷贝数据是CPU-GPU传输通道。

   全局内存提供海量容量（80GB），能容纳大模型的巨大显存，成本相对低（比SRAM便宜100倍）是GPU存储的基础。

2. 第二层：L2缓存

   - 物理位置：GPU芯片内，所有SM共享
   - 容量：40MB（A100）
   - 带宽：约3TB/s
   - 延迟：约200GPU周期（约100ns）
   - 编程控制：自动管理（硬件控制）
   - 可见性：所有SM的所有线程

   可以为全局的数据加速，缓存全局内存的热点数据，数据共享的枢纽，SM之间通过L2缓存交换数据，能保证数据一致性：所有SM看到的L2数据一致

   可以缓解显存的带宽瓶颈，AI训练的绝大部分实际访问同一批的权重，L2缓存命中时可以节省95%的HBM带宽，硬件的自动管理使程序员无需干预，降低了编程复杂度

3. 第三层：L1缓存/共享内存

   - 物理位置：每个SM的内部
   - 容量：192KB/SM（A100，可以配置）
   - 带宽：1TB/SM
   - 延迟：20-40周期（约10-20ns）
   - 编程控制：完全手动（\__shared__）
   - 可见性：Block内所有线程可见

   L1缓存是一个线程协作的仓库，块内线程通过共享内存交换数据（比如矩阵乘法的分块数据），可以手动性能优化，将热点数据显式放入共享内存可以实现接近寄存器的速度，L1缓存功能不手动使用时自动作为L1缓存缓存全局内存的数据

   L1速度比L2快5倍，灵活性高，高程序员可控，实现复杂算法（规约、扫描、卷积等），有成本效益，总容量192KB/SM x 108SM = 20MB，比纯SRAM L2 廉价

4. 第四层：常量内存

   - 物理位置：芯片内专用缓存
   - 容量：64KB
   - 带宽：广播机制（1次读取服务32线程）
   - 延迟：约5周期（命中缓存）
   - 编程控制：只读，\__constant__声明
   - 可见性：全GPU所有线程只读

   可以存储广播数据，同一个Warp中所有线程读取同一常量时（例如学习率）只需要一次内存访问，可以存储配置参数，存储Kernel调用参数、查找表、常量系数

   极致能效，可以节省99%带宽，专用缓存，不占用L1/L2资源

5. 第五层：寄存器文件

   - 物理位置：SM内的每个CUDA核心旁
   - 容量：256KB/SM(A100)
   - 带宽：理论上10TB/SM
   - 延迟：1周期（零开销）
   - 编程控制：完全自动（编译器分配）
   - 可见性：线程私有

   寄存器文件可以零延迟计算，存储线程的局部变量、临时结果，做到极致并行，每个线程255个寄存器，支持深度流水线

   寄存器文件速度快，延迟只有1周期，但是昂贵且容量少，容量限制决定了并行度，寄存器的用量越少，SM驻留的Warp越多

内存类型中的容量越大、速度越慢、价格约便宜，为了在速度和容量以及成本之间权衡，需要上面的分层策略，其中全局内存是通用的仓库，共享内存是专用仓库，通过程序员精确控制。L1/L2是自动缓存。

局部性的原理（空间局部性：相邻数据很可能一起访问；时间局部性：刚访问的数据可能再次访问）也使得GPU需要分层，L2缓存利用时间局部性缓存重复访问的权重，共享内存利用空间局部性手动加载分块数据，Warp利用常量内存的广播特性1次读取服务32线程。

CPU和GPU的本质区别是GPU内存系统为了吞吐量优化容忍高延迟，而CPU内存系统为了延迟优化降低延迟，GPU需要更多的层级和手动的控制。

### TPU架构

TPU是一种ASIC芯片（专用集成电路），为神经网络计算而生。只为了加速张量运算

![image-20260425204616872](C:\Users\32109\AppData\Roaming\Typora\typora-user-images\image-20260425204616872.png)

标量单元起到控制单元的作用，职责类似CPU，向量单元负责执行元素级别的操作，向量存储用来存储需要操作的向量，矩阵乘法单元用来进行矩阵乘法操作（占芯片的最大面积），这些单元构成TPU的Tensor Core，对应了GPU的SM，是一个独立的原子单元。

TPU和GPU区别在于TPU仅仅优化矩阵乘法，不尝试执行通用的计算，架构更加简单，同时TPU中没有Warp，只有块，矩阵乘法和非矩阵乘法的平衡。

张量核心可以处理任意维度的张量，可以进行索引操作。矩阵乘法单元执行的核心计算是矩阵乘法，始终针对张量进行批量矩阵乘法的操作，可以处理张量但是实际进行的是矩阵乘法而非张量运算

GPU成功的原因在于优越的可扩展性，只需要增加SM的数量就可以提升算力，无需担心提高时钟频率带来的散热问题，编程方面由于编程模型的设计每个SM内线程会对不同数据执行相同指令，实际并不复杂，此外GPU中的线程都很轻量级，可以随时暂停或者启动，使得GPU在每个SM中实现极高的利用率

### 性能扩展的趋势

1. 矩阵运算又快又好

   张量核心在V、T系列被引入，专门用于矩阵乘法，矩阵乘法运算比其他的浮点运算快10倍以上，从张量核心被引入后，GPU中矩阵运算和非矩阵运算的速度展现了巨大的差异

2. 计算和内存的扩展不平衡

   互联带宽增长<内存带宽增长<<计算性能增长，未来的发展瓶颈可能是内存带宽的限制，引文内存扩展的速度远低于计算扩展

   ![image-20260425211057023](C:\Users\32109\AppData\Roaming\Typora\typora-user-images\image-20260425211057023.png)

   这是一个屋顶线的模型，横轴代表操作强度，表示计算和数据移动的比率，操作强度很高时计算设备进行大量的计算但是数据移动相对较少，而操作强度低时数据移动占主导。纵轴是吞吐量，表示计算设备每秒可以完成的浮点运算次数。

   不同的线代表不同的内存结构，到达的上限为理想情况下的最大吞吐量。随着操作强度的提升，不同内存层次的吞吐量也会增加，直到到达某个上限，这个上限由内存层次的带宽和延迟特性决定。GPU寄存器由于高速访问的特性，可以在较低的操作强度下达到高吞吐量，但是容量有限，CPU的主内存由于访问速度慢，即使在高强度的操作下吞吐量也相对较低。

   对角线区域存在着内存瓶颈，此时计算能力受限于计算强度（每字节浮点运算数），需要避开左侧的内存受限区域，力求达到右侧的吞吐量受限区域，尽可能减少对低速全局内存的访问频次。

### 性能优化技术

1. **避免串行执行**（不是内存限制的瓶颈）：GPU采用SIMT执行架构，同一线程束Warp内的所有线程要同步执行相同指令，但是操作数据不同，现代GPU中一个Warp通常包含32个线程NVIDIA或64个线程AMD，当Warp内线程遇到分支时，如果线程满足不同的分支便发生了分支发散，GPU使用掩码屏蔽机制串行化处理，先执行if分支，else路径的线程被临时屏蔽，if完成后再执行else分支，执行else分支时if分支被屏蔽，导致一个Warp执行的时间近似等于两条路径耗时之和，计算资源利用率下降。

   所以要尽量避免同一线程束中的线程发

   散，尽量避免条件分支，发散不仅延长执行时间，还可能破坏内存访问的合并性，进一步降低带宽效率。

2. **低精度提升性能**：本质时使用精度换取更快的计算和更省的带宽，背后是硬件电路、存储数据量减少、专用加速单元三重优化叠加。计算数据和权重等所有元素的比特数减少时，需要移动的比特量大幅降低，即便从全局内存访问，其影响也会很小。

   - 提速机制一：硬件因素，浮点运算器的复杂度和位宽成正比，位数越大的浮点运算器体积和复杂程度越大，FP16的乘法器晶体管数量是FP32的1/4，代表可以在同样面积中放入更多浮点运算器，计算能力越强。
   - 提速机制二：内存带宽节省（数据量减少50%），低精度下同参数量的模型文件比高精度占用的内存低，例如同参数模型在FP32下占用的内存是BF16的两倍。在带宽计算上，FP32传输的速度是BF16的1/2，所以低精度下加载模型权重时间成倍减小，缓存的命中率提升（同样的缓存容量。存的数据更多），显存可以容纳更大的模型
   - 提速机制三：Tensor Core专用加速，Tensor Core是NVIDIA为低精度矩阵乘法设计的专用电路，是重构版的矩阵引擎。**加速的原理**：脉动阵列Systolic Array：数据在矩阵中流动，每个周期每个单元完成一次乘加，32x32阵列每周期完成1024次运算，权重静止：矩阵权重预加载到阵列寄存器，减少数据搬运，累加器优化：FP32累加器保证精度，输入输出使用低精度
   - 提速机制四：并行度提升（同样芯片面积，计算单元翻倍）：芯片面积优化，精度越低CUDA核心面积越小，计算能力越强，同时在架构设计上复用寄存器文件，FP32模式64个CUDA核心，每周期64次FMA，FP16模式有64个CUDA核心和4个Tensor Core活跃，每周期64次FMA和1024矩阵运算。同样的芯片面积低精度可以集成4倍的计算单元，实现4倍的吞吐量

   并非所有网络组件和训练算法都能使用低精度处理，混合精度矩阵乘法通常把输入设为16位低精度，乘法运算保持全32位精度，在累加部分和时，中间计算需要高精度保障，因此采用FP32累加器，张量核心最终输出FP32结果，也可以按需降回16位。

3. 算子融合 Operator Fusion

   算子融合通过消除中间结果内存读写和减少Kernel启动开销实现2-5倍的速度提升

   ![image-20260425231644200](C:\Users\32109\AppData\Roaming\Typora\typora-user-images\image-20260425231644200.png)

   从上图来看如果将计算能力增强但是内存的传输带宽有限，导致从内存到计算的传输速率仍然没有改变，提升的那一部分的计算性能将无法得到利用，整体的性能仍然受限于从内存到计算的传输速率。

   在没有进行融合的场景下：一开始的数据需要搬运到计算单元上完成计算，随后将计算的结果数据又搬回到内存单元中，在下一次计算时发现再次需要上一次计算的结果时不得不再次将数据从内存单元搬运到计算单元，这回导致数据在计算单元和内存单元之间来回传输，效率低下的同时会产生巨大的内存开销。

   在右侧的计算流程中发现运算之间不存在依赖关系，所以可以将计算的中间结果始终驻留在计算单元内，这是融合核函数的思维模型，当一系列的运算需要按顺序处理同一数据时，尽可能在单个计算单元完成所有的操作，直到必须传回内存时才进行传输，这是核融合的核心思想。

   ```python
   class GraphModule(torch.nn.Module):
   	def forward(self, x: torch.Tensor):
   		sin = torch.sin(x)
   		pow_1 = sin ** 2
   		cos = torch.cos(x)
   		pow_2 = cos ** 2
   		add = pow_1 + pow_2
   		return (add,)
   ```

   上面编写的神经网络模块在PyTorch中运行时会生成这样的计算图：

   - 载入$x$
   - 启动CUDA核函数计算$\sin x$
   - 启动CUDA核函数计算$\cos x$
   - 计算$\sin x^2$和$\cos x^2$
   - 计算两者加和

   五个运算之间没有复杂的依赖，只需要少量的内存就可以融合为单个运算，在GPU单线程上完成所有的处理，不需要把数据写回全局内存

   本质上算子融合是把多个连续操作合并为单个的CUDA核函数，避免中间结果写回全局内存

4. 重计算 Recomputation

   核心思想是通过增加计算量避免内存访问

   

   ![image-20260425234008228](D:\Typora\image-20260425234008228.png)

   获取输入（最底层的叶子节点）后向上传播激活值，到根节点后反向计算雅可比矩阵，计算梯度时进行反向传播，需要进行乘法运算，将雅可比矩阵和激活值结合反向传播梯度。前向传播中的黄色数值必须存储起来，随后从我们存放的全局内存中提取送入计算单元，但是过程中会导致海量的内存输入输出操作，这种操作可以被避免。

   假设有三个sigmoid函数堆叠在一起，x作为输入，s1、s2、最终输出out作为所有的输出，在前向的计算过程中总共需要三次内存写入和一次内存读出，在反向传播过程中需要调用最终输出out和中间的输出s1s2，最终得到x的梯度，整个过程中总共涉及八次内存的读写操作。

   重计算的核心思想是不存储激活值也不放入内存，而是在反向传播过程中动态重新计算，在新的前向传播中不存储s1和s2，输入x后直接获得输出，前向传播阶段涉及一次内存读取和一次内存写入，反向传播中同时获取out和输入的x进行计算，然后在流处理器本地内存中动态计算每个sigmoid函数，融入反向计算图，这样反向传播涉及两次内存读取和最后的一次内存写入（写入x梯度），总计五次内存读写操作。

   重计算适用于计算成本比读写成本小的场景，无需全局内存读取操作的代价是重新计算三个sigmoid函数，属于使用过剩的计算能力换取内存带宽。

5. 内存合并 Memory Coalescing

   GPU中慢速内存（全局内存/DRAM）极其缓慢，为了提升速度在硬件层面会进行特定的优化。

   一项优化是通过“突发模式”读取——每次读取可以提供多个字节，在读取某一个内存值时获取的是整块内存数据，被称为突发模式，每个地址空间被划分为突发段，系统会直接返回突发段而非仅仅返回目标数据。突发段为了优化耗时的数据迁移过程，如果访问得当可以显著加速内存访问。通过精细涉及内存访问策略，从每个突发段提取所需数据理论上可以实现四倍吞吐量（突发段为4字节），这就是内存合并。

   当线程束中所有线程都位于同一突发段时，智能硬件和编程模型会把这些查询变为单次查询0可以从突发模式DRAM中同时读取所有的四个值，线程束中包含32个有序线程，但是线程的内存访问时同步进行的，经过优化可以一次性访问全部的4字节数据，使得内存的吞吐量提升四倍。

   例如矩阵乘法中每个线程负责矩阵的一整列时同一个Warp中的线程会访问连续的内存地址，可以实现完全的内存合并，从而显著的提升访问效率。

   内存访问顺序的差异属于很底层的优化细节，一旦遍历顺序设计不当，性能会出现数量级的下降。

6. 分块（瓦片）Tiling

   核心思想是通过分组内存访问减少全局内存的访问量。

   ![image-20260426003725024](C:\Users\32109\AppData\Roaming\Typora\typora-user-images\image-20260426003725024.png)

   原始的矩阵乘法存在严重问题，在计算乘积时需要遍历M的行和N的列，内积运算后结果存入P矩阵，右侧时每个线程对应的输出存储位置和元素访问顺序。

   这里的内存访问是非合并的，行矩阵的访问顺序也不连续，存在重复的内存访问，例如$M_{0,0}$被首个线程访问后又被其他线程访问，这些值会从全局内存中被多个线程反复读取，可能导致严重的性能下降。

   理想方案是用一段时间把数据块从全局内存加载到高速的共享内存，在共享内存中完成大量的计算，最后处理下一个数据块，这样能最小化全局内存访问。

   将上面的矩阵分成2x2的小矩阵，首先加载$M_{0,0}$和$N_{0,0}$分块，计算部分求和结果，处理完这两个数据块后加载新的数据块到共享内存中，比如下次加载$M_{0,2}$和$N_{2,0}$重复计算过程，再把部分和累加到P中，减少了全局内存的访问量，同时由于加载的是完整的数据块，可以按照任意顺序（列优先或者行优先）遍历子矩阵，从全局内存加载到共享内存时实现内存访问的合并。带来全方位的性能提升。

   ![image-20260426004713698](C:\Users\32109\AppData\Roaming\Typora\typora-user-images\image-20260426004713698.png)

   例如上图中的NxN规模的矩阵，如果使用非分块矩阵乘法需要从全局内存中读取N次元素，使用分块矩阵乘法时从全局内存中加载读取N/T次数据块（数据块规模为TxT），再在每个数据块中读取T次元素，实际的总的读取次数没有改变，但是由从全局内存中读取N次元素变为从全局内存中读取N/T次元素（数据块）加上从共享内存中读取T次元素（从数据块中读取元素），使得全局内存数据读取量减少了T倍。

   影响瓦片大小的因素主要有合并内存访问、共享内存大小、矩阵维度的可除性，分块的策略很复杂，瓦片大小无法整除矩阵大小时会导致利用率很低，因为有的分块中没有什么数据，对应的分配给瓦片的SM基本处于闲置状态；遇到计算瓶颈时必须优化分块尺寸；必须实现内存访问的合并不能超过共享内存容量；在分块划分不均匀时还会出现突发传输的区段不能和分块对齐，导致突发传输区段错位，这是会导致内存访问量翻倍，可以通过填充padding让矩阵尺寸均匀。

### Flash Attention

Transformer的标准Attention计算在长度为N的序列中具有$O(N^2)$的计算和显存复杂度，性能瓶颈主要来源对高带宽显存HBM的频繁访问而非纯计算开销，由于GPU上kernel调度开销以及不规则的内存访问模式，这类方法在实践中往往难以实现理想的加速效果。

Flash Attention通过分块策略，在片上SRAM中完成分块的$QK^T$、online softmax和V的累积计算，避免将完整的Attention matrix显式并写入HBM，通过显著减少HBM访问次数并提高数据复用率，提升计算密度，实际中将Attention计算从带宽限制转变为接近计算限制，获得显著的性能提升。

不同Attention的实现方法的运行时间差别不大，但是FlashAttention的显存占用明显更低，约为其他实现方法的一半，在长序列场景下有更好的可扩展性和更高的资源利用率。

1. V1计算原理（$Q,K,V \in R^{N\times d}$)\

   ```text
   # 初始化
   m_i = -inf
   l_i = 0
   O_i = 0
   for each Q block i:
       load Q_i
       m_i = -inf      
       l_i = 0         
       O_i = 0         
   
       for each K,V block j:
           if causal and j > i:
               continue
   
           load K_j, V_j
           S_{ij} = Q_i @ K_j^T      # (B_r, B_c)
   
           if causal and i == j:
               apply mask to S_{ij}
   
           # softmax累加 → 归约
           update m_i, l_i, O_i(online softmax)
   
       write O_i to HBM
   ```

   GPU中Attention计算主要瓶颈在缓存(SRAM)容量和计算带宽，直接计算$QK^T$会生成一个NxN的矩阵，N很大时无法一次性放在高速缓存，频繁访问HBM会加大缓存压力。FlashAttention通过分块计算减少对全局内存的访问，提高缓存利用率，保证计算效率和数值稳定性，在SRAM中处理可以容纳的小块，同时进行softmax累加，大幅降低全局内存的访问压力。

2. online softmax

   和标准的softmax在数学上完全等价，都需要全局归一化，不同之处在于online softmax采用流式的计算方式，遍历时动态维护当前的最大值和归一化因子（指数累加和），不存储完整的中间结果，使其和分块计算自然结合，在Flash Attention中的注意力分数按块计算，在块间持续更新最大值和归一化印象实现跨数据块的全局归一化

   ```text
   m_{ij} = rowmax(S_{ij})
   m_new = max(m_i, m_{ij})
   
   l_i = exp(m_i - m_new) * l_i 
     + sum(exp(S_ij - m_new), axis=1)
   
   O_i = exp(m_i - m_new) * O_i 
     + exp(S_{ij} - m_new) @ V_j
   
   m_i = m_new
   O_i = O_i / l_i
   ```

3. Flash Attention V2:面向Tensor Core的并行优化：为了充分发挥现代GPU中矩阵乘法的峰值吞吐能力，在V1的核心思想上实现了更高效的并行和更低的非矩阵乘法开销。

   核心改进是分块并行化处理Q：V1中仍然存在跨块的在线归一化和重标定操作，形成了数据依赖链，计算中表现为matmul->规约（归一化）->标量更新->matmul->规约 & 标量更新的交替模式，非matmul会降低Tensor Core的利用率，限制算子融合和并行调度效率。

   V2 没有改变Q外层，KV内层循环的基本数据加载顺序（单个线程块计算视角下），优化重点在于重构并行策略和规约方式，缓解数据依赖链的性能问题：

   - 从一个$Q_i$对应一个线程块串行扫描所有KV的方式改为对同一$Q_{i}$，在KV维度进行切分，由多个线程块并行处理
   - 保留online softmax数学形式，通过分块并行计算和跨线程块规约的方式，把沿KV维度的串行依赖处理转化为并行计算和末端规约，显著提升并行度和硬件利用率。

   ```text
   for each Q block i in parallel:              # V2的核心：Q 维度并行
       load Q_i into SRAM                      
   
       # 初始化
       m_i = -inf                             
       l_i = 0                                
       O_i = 0                               
   
       for each K,V block j:
           load K_j, V_j into SRAM             
   
           S_ij = Q_i @ K_j^T                  # Tensor Core
           m_ij = rowmax(S_ij)                
           m_new = max(m_i, m_ij)
           
           # 重标定旧累积
           alpha = exp(m_i - m_new)           
           P_ij = exp(S_ij - m_new[:, None])   
           l_ij = rowsum(P_ij)               
   
           # 更新归一化因子
           l_new = alpha * l_i + l_ij         
   
           # 更新输出（核心优化点之一），online softmax累加
           O_i = (alpha[:, None] * O_i) + (P_ij @ V_j)
           m_i = m_new
           l_i = l_new
   
       # 末端归约
       O_i = O_i / l_i[:, None]
       write O_i back to HBM
   ```

   V2提升的本质是使用更多的matmul和更少的串行依赖，而不是单线程内的批量提交的matmul，没有减少操作的数量，将串行的依赖改为了空间并行。

   V2在保持V1内存访问高效特点的情况下显著提升了Tensor Core的利用率，将注意力计算的整体吞吐推向了更高水平，使得长上下文Transformer的训练和推理在单卡或者有限资源缓解下更加高效。

4. V3：异步和低精度优化

   H100（Hopper）架构引入了革命性的新特性：**异步执行模型**（支持计算与数据搬运完全重叠）、**WGMMA指令**以及**原生FP8支持**。V3正是针对这些硬件特性进行深度优化的产物，其目标是让注意力计算在H100上实现**接近理论峰值的吞吐量**，并充分利用低精度格式带来的算力红利。

   核心的改进在异步WGMMA流水线，V3不是建立在Q外循环K/V内循环的顺序变化上，在V2已有的并行和规约框架的基础上，对底层的执行机制重构，引入基于Hopper架构的异步计算能力，把计算流程组织成生产者-消费者流水线

   利用Hopper架构中异步指令实现

   - 数据加载与Tensor Core解耦
   - 跨warp group的流水线并行执行，计算一个block数据的同时预取下一个block的数据

   从而显著减少因为数据等待导致的SM空闲周期，使GPU执行任务的周期更加接近理想的持续计算状态

   ```text
   # 初始化double buffer
   buffer_K[2], buffer_V[2]
   
   for each Q block i:
       load Q_i
   
       async_load(buffer_K[0], buffer_V[0])
   
       for j in range(num_blocks):
   
           curr = j % 2
           next = (j + 1) % 2
   
           # 生产者，提前加载下一块
           if j + 1 < num_blocks:
               async_load(buffer_K[next], buffer_V[next])
   
           # 消费者，使用当前buffer计算
           S_ij = wgmma(Q_i, buffer_K[curr])
   
           update m_i, l_i   # softmax reduce
   
           O_i += wgmma(P_ij, buffer_V[curr])
   
           # 只在必要时同步（避免stall）
           wait_for(buffer_K[next])
   
       write O_i
   ```

   WGMMA(**Warpgroup Matrix Multiply-Accumulate**)是H100引入的新指令，把一组warp组织为warpgroup（4个warp共128个线程），在硬件层面直接支持异步执行，把$Q_iK_j^T$的矩阵乘法拆分成多个WGMMA指令，在Tensor Core执行的同时CPU可以继续发射后续的数据加载指令，实现计算和数据传输的重叠，大幅提高SM的利用率

   V3支持FP8输入，理论上吞吐量可提升2倍，但是需要解决稳定性问题，因为softmax对精度敏感。V3采用混合精度策略提升性能，矩阵乘法使用FP8执行，累加器保持FP16或者BF16精度避免精度损失，softmax使用FP32计算保证指数运算的数值稳定性，输出PV可选择转为FP8适应后续层。

   V3实现了动态缩放因子模型，按块（tile）动态计算缩放因子，并在流水线中传递，确保 FP8 计算的精度与 FP16 相当。

   V3对寄存器实现精细控制，避免寄存器溢出到L1缓存，将中间统计量尽量留在寄存器中，减少了不必要的内存访问，进一步提升流水线效率。

   在性能表现上计算效率、速度、长序列能力都有提升：

   - 计算效率：在 H100 SXM 上，使用 FP8 时，V3 的 TFLOPS 利用率可达 **75%~80%**，接近理论峰值。这意味着注意力计算几乎不再受内存带宽限制，而是完全由计算能力驱动。
   - 速度提升：相比 V2 在 H100 上的 FP16 实现，V3 的 FP8 版本速度提升约 **1.5~2 倍**；即使在相同 FP16 精度下，V3 的异步优化也能带来约 **1.3 倍** 的加速。
   - 长序列能力：在 128k 甚至 1M 长度的序列上，V3 仍能保持极高的吞吐，使得超长上下文模型的训练和推理成本大幅降低。单卡 H100 80GB 可稳定训练 256k 长度的模型。

   V3通过**异步WGMMA流水线**解耦了数据加载与计算，利用**FP8混合精度**释放了H100的低精度算力，并通过精细的**块布局与寄存器优化**消除了内存瓶颈。这一系列改进使得注意力计算不再是Transformer长上下文扩展的阻碍，而是成为能够充分利用硬件极限的“高性能组件”。V3的诞生，直接支撑了当前工业界100k~1M上下文窗口的大语言模型的工程实现。

   V1->V2->V3，从“减少内存带宽瓶颈”到“提升并行调度和GPU利用率”到“实现计算和访存重叠，让SM始终忙碌”

### Page Attention

传统的KV Cache管理方式中通常为请求序列预先分配一段逻辑上连续的缓存空间用来存储历史生成的K/V状态，然而，由于实际生成长度难以精确预测，这种静态预分配策略会带来显存利用率问题。当预留空间大于实际生成长度时，会产生内部碎片（internal fragmentation），即已分配但未被使用的显存空间；同时，由于不同请求的生命周期不一致，释放时间存在差异，显存中可能形成多个零散的空闲块，这个时候尽管总空闲容量充足，但由于无法提供足够大的连续缓存块来满足新序列的分配需求，从而产生外部碎片（external fragmentation）。

碎片问题会降低GPU的HBM利用率，根据操作系统的分页机制把KV Cache拆分成固定大小的页面，通过页表间接地址映射，使逻辑连续的KV存储不再依赖物理连续内存，从而有效减少碎片并提升显存利用率。

核心思路是把KV Cache切分成固定大小的块，将逻辑上连续的token对应的KV映射到物理上未必连续的存储块中，最大化利用显存空间，由于处理好映射关系后先查页表在找到对应物理block保证了读取顺序逻辑正确，所以不需要连续的物理空间，同时内存分配过程动态进行，减少了因提前预留导致的严重内部碎片问题。

映射逻辑表示为

```text
请求 ID + Token 偏移量 -> 逻辑块索引 -> Block Table 检索 -> 物理内存基址 -> 偏移读取
```

**Flash Attention和Page Attention对比**

都致力于提升大模型的推理性能，但是优化维度有本质的不同。

Flash Attention聚焦单次前向计算的IO复杂度，属于算子级的微观优化，解决算的更快的问题。

Page Attention聚焦整个生成生命周期中的显存管理效率，属于系统级的宏观设计，解决“存的高效”的问题。

在影响对象上，FlashAttention 作用于 attention kernel 本身，PageAttention 则管理 KV Cache 的生命周期。此外，FlashAttention 对训练和推理均有影响，而 PageAttention 基本仅适用于推理场景；FlashAttention 不依赖 decoder-only 架构，PageAttention 则对此有强依赖。

实际的推理框架中（例如vLLM）中两者通常协作使用，同时优化单请求延迟和整体吞吐量。但是由于Page Attention使用了间接寻址的非连续物理布局，可能破坏FlashAttention 对连续内存访问与 memory coalescing 的假设，使用时需要以page为最小的streaming计算单元适配，或者确保block_size和tile_size相近，来维持共享内存利用率和访存效率。
